# 旅游规划 Agent — 技术方案分析

> 项目定位：个人本地使用的 Python 旅游规划助手
> 目标：1-2 周快速验证原型（MVP）
> 技术栈：Python 优先

---

## 一、拆解阶段（Decomposition）

### 模块一：需求采集与理解（输入层）

**阶段1：用户需求采集**
- **挑战1**：用户输入自由散漫，目的地/时间/预算等关键信息缺失或模糊
  - **策略**：结构化对话引导
  - **方法**：LLM 驱动的多轮对话，逐项询问目的地、出发日期、天数、预算、同行人、兴趣偏好（美食/自然/人文/购物等）
- **挑战2**：兴趣偏好过于笼统（如"随便看看""好吃就行"）
  - **策略**：选项引导 + 举例
  - **方法**：提供常见兴趣标签让用户选择，配合示例说明
- **挑战3**：用户同时有多个目的地或灵活行程
  - **策略**：灵活行程模式
  - **方法**：支持"出发地+时间范围+兴趣"的开放式输入

**阶段2：需求结构化**
- **挑战1**：自然语言需求转为结构化查询参数
  - **策略**：LLM 信息抽取
  - **方法**：用 LLM 将对话内容抽取为结构化 JSON（destination, dates, budget, interests, etc.）
- **挑战2**：需求冲突或不可行（如预算过低、时间不足）
  - **策略**：可行性预检
  - **方法**：LLM 评估需求合理性，给出调整建议

---

### 模块二：信息采集（搜索层）

**阶段1：景点信息获取**
- **挑战1**：景点数据来源分散，无统一 API
  - **策略**：多渠道聚合
  - **方法**：Web 搜索 + 旅游网站爬取（携程/马蜂窝/小红书）+ 维基百科
- **挑战2**：信息时效性（营业时间、门票价格变动）
  - **策略**：实时搜索优先
  - **方法**：优先通过搜索引擎获取最新信息，标注信息获取时间
- **挑战3**：海外旅游信息获取难度大
  - **策略**：多语言搜索
  - **方法**：用 LLM 翻译搜索关键词，获取本地语言信息后用 LLM 翻译回中文

**阶段2：餐饮信息获取**
- **挑战1**：餐厅推荐需要个性化（口味、预算、排队情况）
  - **策略**：多维度筛选+实时搜索
  - **方法**：通过搜索引擎/点评网站获取餐厅列表，结合用户偏好筛选
- **挑战2**：餐厅营业时间、预订信息不准确
  - **策略**：信息来源标注
  - **方法**：显示信息时标注来源和获取时间，提示用户以实际为准

**阶段3：住宿推荐**
- **挑战1**：住宿价格实时波动
  - **策略**：价格区间+比价引导
  - **方法**：提供大致价格区间和平台推荐，引导用户自行比价
- **挑战2**：位置与行程的匹配度
  - **策略**：地理相关性优先
  - **方法**：推荐住宿时考虑与规划景点的地理位置接近程度

**阶段4：交通与路线**
- **挑战1**：城际交通信息实时变化
  - **策略**：班次查询+外部链接
  - **方法**：查询大致班次信息，提供 12306/携程等外部链接
- **挑战2**：市内交通耗时估算
  - **策略**：距离+交通方式估算
  - **方法**：基于地理位置计算距离，按步行/公交/打车估算耗时

**阶段5：天气与实用信息**
- **挑战1**：天气预报只有短期准确
  - **策略**：季节气候参考+临期提醒
  - **方法**：提供历史同期气候参考，临期建议用户查看天气预报
- **挑战2**：签证/证件/货币等实用信息
  - **策略**：LLM 常识库 + 官方来源
  - **方法**：利用 LLM 知识提供通用建议，标注"建议以官方信息为准"

---

### 模块三：行程规划引擎（核心层）

**阶段1：日程生成**
- **挑战1**：每天行程过多或过少，节奏不合理
  - **策略**：合理密度控制
  - **方法**：每天安排 3-5 个活动点，留出用餐和休息时间
- **挑战2**：景点间路线不合理，来回折返
  - **策略**：地理聚类排序
  - **方法**：按地理位置将景点分组，同区域安排在相邻时段
- **挑战3**：开放时间冲突（到景点才发现关门）
  - **策略**：时间约束检查
  - **方法**：在排行程时检查各景点的开放时间/休息日
- **挑战4**：用户不同兴趣的权重分配
  - **策略**：兴趣加权分配
  - **方法**：根据用户兴趣标签分配每天的活动类型比例

**阶段2：多方案生成**
- **挑战1**：单一方案无法满足用户所有偏好
  - **策略**：多方案并行生成
  - **方法**：生成 2-3 种不同风格的行程方案（如"经典打卡版""深度体验版""休闲放松版"）
- **挑战2**：不同方案间差异不够明显
  - **策略**：差异化策略
  - **方法**：每种方案设定不同的优先级策略（效率优先/体验优先/性价比优先）

**阶段3：预算管理**
- **挑战1**：总花费超出用户预算
  - **策略**：预算跟踪与调整
  - **方法**：在生成过程中实时计算预计花费，超预算时自动替换为低价替代方案
- **挑战2**：隐性消费（交通、保险等）
  - **策略**：费用全面估算
  - **方法**：门票/餐饮/住宿/交通/其他分类估算

---

### 模块四：交互与输出（展示层）

**阶段1：行程展示**
- **挑战1**：行程信息密集，阅读体验差
  - **策略**：结构化展示
  - **方法**：按天分组 → 按时间排序 → 每个活动含名称/描述/地址/预估花费/建议时长
- **挑战2**：缺少地图可视化
  - **策略**：简易地图标注
  - **方法**：为每个地点标注坐标，可在地图上标注（适合时集成 folium/leaflet）

**阶段2：交互调整**
- **挑战1**：用户想微调行程但不希望重新生成
  - **策略**：局部修改
  - **方法**：支持用户对特定天/特定活动进行调整，LLM 重新优化
- **挑战2**：用户想对比不同方案
  - **策略**：方案对比视图
  - **方法**：并排展示不同方案，标注各自特点

**阶段3：行程导出**
- **挑战1**：用户希望离线保存
  - **策略**：多格式导出
  - **方法**：支持导出为 Markdown / PDF / 纯文本 / iCal 格式

---

### 模块五：记忆与个性化（持久层）

**阶段1：历史记录**
- **挑战1**：用户重复查询同一目的地
  - **策略**：本地缓存 + 更新提醒
  - **方法**：缓存历史行程，提示用户之前查过，可复用或更新
- **挑战2**：用户偏好跨会话记忆
  - **策略**：本地偏好文件
  - **方法**：保存用户偏好到本地 JSON 文件（预算偏好、兴趣倾向、出行习惯）

## 二、枚举（Options）

### 关键决策点1：信息采集方式

| 方案 | 描述 |
|------|------|
| **方案 A：纯 LLM 知识** | 仅依赖 LLM 内部知识生成旅游信息，不上网搜索 |
| **方案 B：LLM + 实时搜索** | 利用搜索引擎 API 获取实时信息，LLM 整理 |
| **方案 C：LLM + 专用 API** | 调用旅游专用 API（如 Google Places、TripAdvisor、Amadeus） |

### 关键决策点2：行程规划引擎实现方式

| 方案 | 描述 |
|------|------|
| **方案 A：LLM 直接生成** | 将信息拼成上下文，让 LLM 直接输出行程 |
| **方案 B：LLM + 算法规划** | LLM 推荐内容 + 约束求解/遗传算法做优化排布 |
| **方案 C：LLM + Template** | 预设行程模板，LLM 按模板填充内容 |

### 关键决策点3：交互方式

| 方案 | 描述 |
|------|------|
| **方案 A：CLI 命令行** | 终端交互，轻量快速 |
| **方案 B：Web UI** | 简单网页界面，Flask/Streamlit |
| **方案 C：GUI 桌面应用** | Tkinter/PyQt 桌面应用 |

### 关键决策点4：数据持久化

| 方案 | 描述 |
|------|------|
| **方案 A：纯文件存储** | JSON/YAML 文件保存行程和偏好 |
| **方案 B：SQLite** | 轻量数据库，支持结构化查询 |
| **方案 C：NoSQL** | 用 TinyDB 等文档数据库 |

## 三、深度分析（Analysis）

### 针对方案一（信息采集方式）

#### 方案 A：纯 LLM 知识

| 项目 | 内容 |
|------|------|
| ✅ 优点 | 实现最简单，无需外部 API，0 外部依赖，响应极快 |
| ❌ 缺点 | 信息可能过时（LLM 知识截止日期前），门票价格/营业时间等变化信息不准确，无法获取小众/新兴景点 |
| ⚠️ 潜在风险 | 用户到了实地发现信息不对，信任度下降；热门景点排队时间等信息缺失 |

#### 方案 B：LLM + 实时搜索

| 项目 | 内容 |
|------|------|
| ✅ 优点 | 信息较新，覆盖面广，能获取实时价格/营业时间/排队情况/用户评价 |
| ❌ 缺点 | 需要搜索 API 成本（或自建搜索），搜索质量依赖搜索引擎，响应较慢 |
| ⚠️ 潜在风险 | 搜索 API 调用费用（如 SerpAPI/Google Search API）；搜索结果噪声多需要 LLM 过滤 |

#### 方案 C：LLM + 专用 API

| 项目 | 内容 |
|------|------|
| ✅ 优点 | 数据结构化好，信息准确度高，有评价/评分等结构化数据 |
| ❌ 缺点 | 大部分旅游 API 需要申请且收费，国内覆盖有限（尤其针对中国旅游场景），集成复杂度高 |
| ⚠️ 潜在风险 | API 政策变更频率高，依赖第三方平台可用性 |

---

### 针对方案二（行程规划引擎）

#### 方案 A：LLM 直接生成

| 项目 | 内容 |
|------|------|
| ✅ 优点 | 实现简单，LLM 对行程安排有较好的常识理解，输出自然流畅 |
| ❌ 缺点 | 可能忽略时间/距离约束，路线优化不够精细，每次生成结果不一致 |
| ⚠️ 潜在风险 | 行程可能过于理想化（忽略实际交通时间），用户到现场发现不合理 |

#### 方案 B：LLM + 算法规划

| 项目 | 内容 |
|------|------|
| ✅ 优点 | 路线和时间安排最优，约束满足度高，可解释性强 |
| ❌ 缺点 | 实现复杂度高（需要地理编码/距离矩阵/LBS 算法），原型周期长 |
| ⚠️ 潜在风险 | 过度工程化，快速验证阶段投入产出比低 |

#### 方案 C：LLM + Template

| 项目 | 内容 |
|------|------|
| ✅ 优点 | 在简单和准确之间取得平衡，模板保证结构一致性，LLM 填充保证灵活度 |
| ❌ 缺点 | 模板固定导致创新受限，不同目的地可能需要不同模板 |
| ⚠️ 潜在风险 | 模板维护成本随目的地增多而上升 |

---

### 针对方案三（交互方式）

#### 方案 A：CLI 命令行

| 项目 | 内容 |
|------|------|
| ✅ 优点 | 开发最快，0 前端工作，适合开发者和技术用户 |
| ❌ 缺点 | 非技术用户使用门槛高，无法展示地图等可视化内容，交互体验差 |
| ⚠️ 潜在风险 | 项目验收时可能被认为"不够完整" |

#### 方案 B：Web UI（Streamlit/Flask）

| 项目 | 内容 |
|------|------|
| ✅ 优点 | Streamlit 可极快搭建原型，支持地图/表格/图表，交互友好，本地运行简单 |
| ❌ 缺点 | 需要额外的前端开发（即使是 Streamlit），部署略繁琐 |
| ⚠️ 潜在风险 | Streamlit 性能有限，但原型阶段足够 |

#### 方案 C：GUI 桌面应用

| 项目 | 内容 |
|------|------|
| ✅ 优点 | 界面原生，用户体验一致，离线可用 |
| ❌ 缺点 | 开发周期长，调试麻烦，Python 桌面 GUI 生态体验一般 |
| ⚠️ 潜在风险 | 快速验证阶段完全不推荐 |

---

### 针对方案四（数据持久化）

#### 方案 A：纯文件存储

| 项目 | 内容 |
|------|------|
| ✅ 优点 | 零依赖，实现简单，可直接用 markdown/json 文件，用户可读可改 |
| ❌ 缺点 | 不支持复杂查询，并发写入有风险，数据量大时管理困难 |
| ⚠️ 潜在风险 | 个人使用场景下几乎无风险 |

#### 方案 B：SQLite

| 项目 | 内容 |
|------|------|
| ✅ 优点 | 支持 SQL 查询，数据结构化好，Python 内置支持，单文件部署 |
| ❌ 缺点 | 需要定义 schema，小型个人项目可能过度设计 |
| ⚠️ 潜在风险 | 原型阶段增加不必要的复杂度 |

#### 方案 C：NoSQL

| 项目 | 内容 |
|------|------|
| ✅ 优点 | 灵活 schema，文档结构直观 |
| ❌ 缺点 | 额外依赖，个人场景优势不明显 |
| ⚠️ 潜在风险 | 小众库可能维护不佳 |

## 四、场景匹配与建议（Recommendation）

### 信息采集方式 → 推荐：方案 B（LLM + 实时搜索）

| 考量 | 说明 |
|------|------|
| **目标优先级** | 原型阶段需要平衡"信息质量"和"实现速度"。纯 LLM 知识信息过时是旅游规划的死穴（用户到了发现景点关闭/价格不对），但专用 API 又太重。实时搜索是最佳折中 |
| **快速验证** | 可以用免费的搜索方案（如 DuckDuckGo 免费 API 或自建爬虫），0 成本启动 |
| **扩展性** | 后期可以随时升级到专用 API，方案 B 的架构天然支持多源接入 |

### 行程规划引擎 → 推荐：方案 C（LLM + Template）

| 考量 | 说明 |
|------|------|
| **目标优先级** | 快速原型优先。LLM 直接生成（方案 A）虽然最快，但质量不可控。算法规划（方案 B）太慢。**LLM + Template** 是黄金平衡点 |
| **实现方式** | 预设行程模板（如"3 天城市游""5 天深度游"），LLM 根据采集到的信息填充模板，保证结构合理性的同时保留灵活度 |
| **退出策略** | 如果项目做大，模板可以升级为算法规划，LLM 填充的部分可以逐步硬化 |

### 交互方式 → 推荐：方案 B（Web UI - Streamlit）

| 考量 | 说明 |
|------|------|
| **目标优先级** | 旅游规划天然需要可视化（地图、时间轴、对比），CLI 无法满足。Streamlit 可以在 1 天内搭出可用的 UI，且有丰富的组件（st.map、st.dataframe 等） |
| **个人使用** | `streamlit run` 即可在本地启动，足够轻量 |
| **后续演进** | 如果需要，可以过渡到 Flask/FastAPI + React |

### 数据持久化 → 推荐：方案 A（纯文件存储）

| 考量 | 说明 |
|------|------|
| **目标优先级** | 个人本地使用，文件存储完全够用。JSON 存偏好，Markdown 存行程，用户直接用文本编辑器也能改 |
| **不要过度设计** | 原型阶段不需要数据库。等用户量大了再说 |

### 总结推荐架构

```
用户输入 (CLI/Streamlit)
    │
    ▼
LLM 需求理解 → 结构化需求 JSON
    │
    ▼
信息采集层 (搜索引擎 API + LLM 提取)
    │  ├── 景点信息
    │  ├── 餐饮信息
    │  ├── 住宿参考
    │  └── 交通/天气
    │
    ▼
行程规划引擎 (LLM + Template)
    │  ├── 每日日程生成
    │  ├── 路线优化
    │  └── 多方案生成
    │
    ▼
输出展示 (Streamlit Web UI)
    │  ├── 按天展示
    │  ├── 地图标注
    │  ├── 方案对比
    │  └── 导出 Markdown/PDF
```

## 五、验证体系（Verification）

### 核心指标

| 指标 | 定义 | 验证逻辑 |
|------|------|----------|
| **行程可用率** | 生成的行程中"实际可执行"的活动比例（非理想化） | 关键验证：抽查生成的行程，检查景点开放时间与安排是否冲突、交通时间是否合理 |
| **信息准确率** | 行程中的景点/餐厅等信息是否真实存在且描述准确 | 随机抽样 10 个生成结果，手动验证准确性 |
| **用户采纳率** | 用户是否直接使用生成的行程，还是做了大量修改 | 记录用户对行程的修改次数，越低说明质量越高 |
| **生成成功率** | 输入需求后成功输出完整行程的比例 | 技术指标，测试 20 组不同输入，统计完整输出率 |
| **端到端延迟** | 从输入需求到看到完整行程的时间 | 原型阶段目标：≤ 30 秒 |

### 验证方法

**自动化测试**：
- 写 5-10 组测试用例（不同目的地、天数、预算），自动化运行并检查输出格式完整性
- 用 LLM-as-Judge 评估行程质量（让另一个 LLM 评价行程的合理性）

**手动验证**：
- 对每个核心目的地（如北京、东京、曼谷），实际验证生成行程的可执行性

**是否需要为以上指标设计自动化测试脚本？** 等你确认后再进入实现细节。

## 六、风险预案（Mitigation & Debugging）

### 风险预防

| 风险 | 预防措施 |
|------|----------|
| 搜索引擎 API 费用超标 | 优先用 DuckDuckGo 免费 API；设置每日调用上限；缓存搜索结果减少重复请求 |
| LLM API 调用成本高 | 使用缓存机制（同一目的地/问题的结果复用）；选择性价比高的模型（如 Claude Haiku/GPT-4o-mini） |
| 行程生成质量不可控 | 引入 LLM 自检步骤（让 LLM 自己审查生成的行程，指出不合理之处并修正）；多次生成取最优 |
| 信息过时 | 所有信息标注获取时间；行程中提示"建议出行前再次确认" |

### 故障修复（Bad Case 归因分析）

| Bad Case | 归因 | 修复手段 |
|----------|------|----------|
| 推荐的景点已经关门/停业 | 信息采集阶段失败——搜索引擎返回了过时信息 | 增加多源交叉验证；标注信息获取日期；提示用户自行确认 |
| 行程路线不合理（折返跑） | 引擎阶段失败——LLM 忽略了地理位置约束 | 引入简单的地理聚类（按区域分类景点再排序）；或给 LLM 提供地点坐标信息 |
| 预算严重超支 | 引擎阶段失败——LLM 没有实时计算累计花费 | 在生成过程中加入预算累计检查步骤，超限时自动替换 |
| 输出格式乱/缺失信息 | 输出阶段失败——LLM 输出未遵循模板 | 增加输出格式校验（JSON Schema / Pydantic）；校验不通过则重新生成 |
| 用户说"不喜欢这个方案"但说不清原因 | 交互阶段失败——没有足够引导用户表达偏好 | 增加结构化反馈入口（评分/标签选择）；LLM 追问具体不满意的点 |

### 典型场景 RCA

**场景：用户想去北京 3 天，生成了包含故宫、长城、颐和园、天坛、南锣鼓巷、三里屯的行程**

可能问题：
1. 第一天上午故宫（合理），下午天坛（合理），晚上三里屯（合理）→ **通过**
2. 第三天安排了长城 + 颐和园 + 圆明园 → **可能过满**，长城来回就要半天
   - **根因**：LLM 低估了长城的地理距离和游玩耗时
   - **修复**：在 Prompt 中明确标注各景点的大致游玩时长和距市中心距离

## 七、优化方向（Optimization）

如果上线后核心指标不达标，按以下方向迭代：

| 优先级 | 优化方向 | 针对问题 | 具体手段 |
|--------|----------|----------|----------|
| P0 | **缓存 + 结果复用** | LLM API 成本高、响应慢 | 对同一目的地的搜索结果缓存 7 天；用户调整行程时只重新生成被修改部分 |
| P0 | **多方案对比增强** | 用户选择困难 | 每个方案标注"适合人群"标签（"适合拍照党""适合吃货""适合省钱党"） |
| P1 | **地理编码集成** | 路线合理性不足 | 引入 Google Maps / 高德 API 获取精确坐标和距离矩阵，替代 LLM 估算 |
| P1 | **用户反馈闭环** | 行程质量提升慢 | 每段行程结束后记录用户实际去了哪些地方，反向训练偏好模型 |
| P2 | **多语言支持** | 海外旅行信息不足 | 用 LLM 翻译搜索词和搜索结果，获取当地语言信息 |
| P2 | **协作分享** | 单人使用场景扩展 | 支持导出行程分享链接，支持多人共同编辑行程 |
